 Fix missing 'equivalenceKey' warning for code fixes
 ### Description

When we create a `CodeAction` for code fix, we don't provide `equivalenceKey` argument. That raises compiler warning that we need to fix:

>RS1010: Provide an explicit argument for optional parameter 'equivalenceKey', which is non-null and unique across all code actions created by this fixer.

This compiler warning is now suppressed in `.editorconfig` to avoid noise.

### Repro steps

Remove RS1010 supression from `.editorconfig` and rebuild the project.

 Deterministic Source Paths Breaks Coverage
 ### Description

If deterministic source paths are enabled during the builds, the scanner will fail to find the code coverage for the code.  (The coverage reports will be indexed, but it will say 0 files matched).  This happens by default with the Microsoft.SourceLink package.

### Repro steps

Create a dotnet core project with 1 class and 1 unit test project, add the Microsoft.SourceLink package, and then run the sonar scanner against it.  It will fail to produce code coverage.

### Known workarounds

Turn deterministic source paths off.

### Related information
* see https://github.com/dotnet/sourcelink/issues/615
* see https://github.com/dotnet/roslyn/issues/43476
* C#/VB.NET Plugins Version - 8.3.1
* Visual Studio Version - N/A
* MSBuild / dotnet version - Core 3.1
* If running through the Scanner for MSBuild, its version - dotnet-scanner 4.8
* Operating System - Win 10.

 Analyzers incorrectly caches rule parameters from the SonarLint.xml file
 ### Description

Rule parameters are passed to the analyzers via a file called `SonarLint.xml` using the Roslyn `AdditionalFiles` mechanism. Roslyn passes the additional files as part of the analysis context each time an analysis is triggered. The contents of the additional files can change between analysis runs.

The sonar-dotnet analyzers load the settings once per AppDomain and then ignore any settings that are supplied in subsequent analysis runs in the same AppDomain. This can lead to the wrong rule parameters being applied.

### Repro steps

1. Open a project in VS and install the SonarC# analyzer (or use SonarLint).
2. Add a `SonarLint.xml` settings with the content given below, with the `Build Action` set to `C# analyzer additional file` (or set the item type to `AdditionalFiles` if you are editing the project file directly.
3. Edit the ruleset to turn on rule S1451
4. Edit a code file to add a copyright header with the wrong year, and check the issue is raised.
4. Change the year in the copyright in the `SonarLint.xml` file so the rule should not fire, and save the file.
5. Edit the code to trigger the rule again.

### Expected behaviour
The new settings are picked up and the issue is not raised.

### Actual behaviour
The new settings are ignored until VS is restarted.

#### Repro SonarLint.xml file
```
<?xml version="1.0" encoding="UTF-8"?>
<AnalysisInput>
  <Settings/>
  <Rules>
    <Rule>
      <Key>S1451</Key>
      <Parameters>
        <Parameter>
          <Key>headerFormat</Key>
          <Value>// Copyright ACME 2020
</Value>
        </Parameter>
        <Parameter>
          <Key>isRegularExpression</Key>
          <Value>false</Value>
        </Parameter>
      </Parameters>
    </Rule>
  </Rules>
  <Files>
  </Files>
</AnalysisInput>
```

### Known workarounds

Restart VS.

### Related information

* C#/VB.NET Plugins Version: all versions with parameterised rules
* Visual Studio Version: all

### Background
The parameters are loaded and cached by the static `ParameterLoader` class that holds a static list of the parameterised analyzers it has processed. If it has already set the parameters for a particular analyzer it won't set them again - [see here](https://github.com/SonarSource/sonar-dotnet/blob/25c1ce8c65d14f2d284657a920dfe2490a3a93b7/sonaranalyzer-dotnet/src/SonarAnalyzer.Common/Helpers/ParameterLoader.cs#L88).

In VS, this means that the once the analyzers have loaded a `SonarLint.xml` file, those rule parameters will be used until VS is restarted. It doesn't matter if the user edits the settings file, switches to a solution with a different `SonarLint.xml` file, or even a different solution that does not have a `SonarLint.xml` file.

I haven't checked whether command line builds are impacted, but my guess is that they are. MSBuild can reuse nodes to build subsequent projects, which might have different (or no) `SonarLint.xml` settings.

Also, both `dotnet` and `MSBuild` have shared build server processes that keep running after a build is complete that can then be used when building other solutions.



 Rule S4790: its content should be replaced by S2070
 For hashing algorithms, several rules exist, in particular these two:
- A security-hotspot: https://jira.sonarsource.com/browse/RSPEC-4790
- That supersedes this vulnerability: https://jira.sonarsource.com/browse/RSPEC-2070

It's not possible to maintain two rules on exactly the same subject, for us and the end users, so:
- S2070 will be deprecated
- the content/implementation of S2070 is more relevant than S4790 because S4790 raises everywhere a hash function is used (even when secure hash function, like SHA-256 is used) and S2070 raises only when a weak hash function is used (like MD5)
- so the content/implementation of S2070 should "be moved" to S4790 (the key of the rule should be updated in Sonar-dotnet), because the type of issue (hotspot) is more relevant

Maybe you can avoid raising the warning when the field is a ref parameter:

Hi @MarcosMeli ,

Thank you for reporting this interesting case. We can confirm it as False Positive. 

Problem here is multithreading AND `ref` keyword. Singlethreaded situation with `ref` keyword should be handled normally for True Positives. 

We'll keep this in our backlog to be aware of the issue, but we don't have clear idea how it should be fixed now due to quite big synchronization possibilities in `System.Threading` namespace and it's derivatives.

Thank you  @pavel-mikula-sonarsource 

I completely agree with you, one option could be to validate if the "always true" expression contains a ref parameter there is no need to raise the warning.
I don't know the internals but just my 2 cents
